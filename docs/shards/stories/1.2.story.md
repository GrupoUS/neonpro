# Story 1.2: Role-Based Permissions Enhancement

## Status
Draft

## Story
**As a** clinic owner,
**I want** to define granular permissions and role hierarchy for my staff,
**so that** I can control access to sensitive patient data and clinic operations based on job responsibilities.

## Acceptance Criteria

1. Define four role levels: Owner (full access), Manager (operational access + reporting), Staff (limited operational), Patient (self-service only)
2. Implement granular permissions system with 20+ specific permissions (patient CRUD, financial access, reporting, etc.)
3. Create role hierarchy where higher roles inherit lower role permissions automatically
4. Implement permission validation middleware that checks every API request
5. Build role management interface for owners to assign/modify staff roles
6. Add role-based UI rendering - hide/show features based on user permissions
7. Create permission groups for easier management (Clinical, Financial, Administrative, Reports)
8. Implement temporary role elevation for specific tasks (e.g., Manager → Owner for 1 hour)
9. Add permission audit trail - track who changed what permissions when
10. Support role-based data filtering (e.g., Staff only sees their assigned patients)

## Dev Notes

### Technical Architecture Context
[Source: architecture/tech-stack.md] - **Authentication System**: Built on Supabase Auth with custom RBAC extensions. Extend existing JWT token to include role and permission information.

[Source: architecture/coding-standards.md] - **Component Architecture**: Use Server Components for role management pages with Client Components for interactive role assignment interfaces.

### Role-Based Access Control Infrastructure
[Source: epic-1-enhanced-authentication.md] - **RBAC Integration**: Part of Enhanced Authentication system. Must integrate seamlessly with MFA system from Story 1.1.

**Role Hierarchy Design**:
- Owner: Full system access + user management + clinic settings
- Manager: Operational access + reporting + staff scheduling
- Staff: Limited operational access + assigned patient management
- Patient: Self-service portal + own data access only

### Database Schema Requirements
[Source: architecture/03-data-model-rls-policies.md] - **RLS Policies**: All role tables must implement Row Level Security for multi-tenant access control.

**RBAC Data Models Required**:
- roles (id, name, level, clinic_id, permissions_inherited)
- permissions (id, name, category, description, resource_type)
- role_permissions (role_id, permission_id, granted_at)
- user_roles (user_id, role_id, clinic_id, granted_by, expires_at)
- permission_audit_logs (user_id, permission_id, action, changed_by, timestamp)

### Permission Categories & Examples
**Clinical Permissions**: patient.read, patient.create, patient.update, patient.delete, medical_history.read, treatment.create
**Financial Permissions**: invoice.read, invoice.create, payment.process, financial_reports.view, pricing.modify
**Administrative Permissions**: staff.manage, clinic.settings, schedule.manage, inventory.control
**Reporting Permissions**: reports.patient, reports.financial, reports.operational, analytics.view

### File Structure & Component Locations
[Source: architecture/coding-standards.md] - **Component Locations**:
- `app/admin/roles/` - Role management pages
- `components/admin/roles/` - Role management components
- `lib/auth/rbac.ts` - RBAC utility functions and middleware
- `types/rbac.ts` - Role and permission TypeScript interfaces
- `hooks/use-permissions.ts` - Permission checking React hooks
- `middleware/rbac.ts` - API route protection middleware

### Integration Requirements
**MFA Integration**: Must work with MFA system from Story 1.1. Higher privilege roles require MFA enforcement.
**Session Management**: Integration point for Story 1.4 - role information in session management.

### Performance Requirements
[Source: epic-1-enhanced-authentication.md] - **Performance Standards**: Permission checks must not add >50ms to API response time. Use caching for frequent permission lookups.

### LGPD Compliance Requirements
[Source: epic-1-enhanced-authentication.md] - **Compliance Standards**: All role and permission changes must be logged with audit trail. Implement data access transparency based on roles.

### Security Framework
**Zero-Trust Verification**: Every API call must validate permissions. No client-side permission enforcement only - always validate server-side.

### Testing Standards
[Source: architecture/coding-standards.md] - **Testing Requirements**:
- Unit tests for permission validation logic

- Security tests for permission bypass attempts
- UI tests for role-based feature rendering
